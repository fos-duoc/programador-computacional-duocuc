### Push y rebase forzoso por divergencia
Al pushear vino la primera sorpresa: el remote rechazó porque el OAuth token de gh no tenía scope `workflow` (lo necesita para crear `.github/workflows/ci.yml`). Resolución: `gh auth refresh -s workflow` con device flow.

Segunda sorpresa: `gh pr create` falló con "branch has no history in common with main". Investigación: el `origin/main` tiene los mismos mensajes de commit pero SHAs distintos al `main` local (`5d15e2c1` vs `406f55d2`) — la history del remote fue reescrita en algún punto (probablemente filter-repo para limpiar archivos grandes), sin merge base. Solución:
1. Crear backup branch.
2. `git rebase --onto origin/main main` → 8/8 commits aplicados sin conflictos (los archivos eran idénticos a pesar del SHA distinto).
3. `git push --force-with-lease` para sustituir el branch remoto con la versión rebaseada.
4. Verificar lint/build post-rebase: ✅ ✅.

### Aprendizajes adicionales (parte 2)

- **Truco "rename puro + modificación"**: cuando `prettier --write` ya divergió los archivos del original, git no detecta renames (umbral 50% por defecto). Para preservar history: extraer HEAD content con `git archive HEAD <path> | tar -x`, sustituir el contenido divergido temporalmente, commit del rename puro, restaurar el contenido divergido, segundo commit con las modificaciones. Resultado: 86 + 932 renames detectados limpiamente.
- **Token gh con scope `workflow`**: NO incluido por defecto en `gh auth login`. Hay que `gh auth refresh -s workflow` antes de pushear cualquier rama que toque `.github/workflows/`.
- **Repos con history reescrita**: el remote `main` y el local `main` pueden divergir si alguien corrió filter-repo. `git merge-base` vacío es la señal. Solución: rebase --onto al remote, no merge.
- **`git push --force-with-lease`** > `--force` siempre que sea posible (verifica que el remote no haya cambiado entre tu fetch y tu push).

### Opción B — Tech Debt Sprint

**Causa raíz de los 7 archivos en el ignore:** dos patrones distintos.

1. **HTML comments dentro de expresiones JSX** (`{...map(() => ...)}`): el parser de prettier-plugin-astro y eslint-plugin-astro no acepta `<!-- ... -->` dentro de un contexto de expresión, sólo dentro del template estático. Afectaba a `CapstoneProjects.astro`, `LearningRoadmap.astro`, `NotionEmbed.astro`, `TechStack2025.astro`, `trayectorias.astro`. Fix: convertir a `{/* ... */}` dentro del JSX.

2. **Caracteres `>` literales en HTML** que el parser interpreta como JSX: en `laboratorios.astro` el `<h3>Consistencia > Intensidad</h3>` y en `DataPipeline.astro` los snippets de código Python (`extract >> transform >> load`, `if (x > 10)`, `if (x >= 10)`). Fix: escapar como `&gt;`.

Una vez fixeados los parser errors, removí los 7 del `.prettierignore` y del `eslint.config.mjs ignores`. Después corrí `eslint . --fix` que arregló 60 warnings (`prefer-const`, `no-var`, `no-unused-vars` con prefijo `_`, etc.) y `prettier --write .` para normalizar formato.

### Aprendizajes (parte 3)

- **HTML comments en Astro**: válidos en el template estático (la parte fuera de `{...}`), pero NO en expresiones JSX dentro de `{...map(...)}` o ternarios. Convertir a `{/* ... */}` solo cuando estás dentro de una expresión.
- **Tokens `>` literales en HTML text**: el parser de eslint-plugin-astro es más estricto que el browser y los toma como JSX. Escapar siempre como `&gt;` cuando es contenido de texto, o usar `{">"}` cuando es valor de atributo o expresión.
- **`vite.build.emptyOutDir: false`**: necesario solo cuando el outDir es compartido con archivos no generados. Mejor migrar a un outDir limpio y resolver la causa raíz.
- **`actions/deploy-pages@v4`** + **`actions/upload-pages-artifact@v3`**: combinación estándar para deploy de sitio estático a GH Pages desde Actions. Requiere `permissions: pages:write, id-token:write` y un job dedicado al deploy con `environment: github-pages`.

- **Hooks locales con `cd`**: si el hook hace cd antes de correr el comando, los paths del pathspec llegan con prefijo desde la raíz; hay que strippearlos.
- **`conventional-pre-commit`** acepta types como argumentos posicionales, NO como `--types`.
- **astro-icon puede estar instalado y configurado pero no usado**. Verificar siempre con `grep -rE "<Icon\b" src/` antes de tree-shake — quizás removerlo entero.
- **Astro 5 `prefetch`** es built-in en config, no necesita integration package.
- **Astro `<style>` scoped purga reglas no matched**: el `@media (prefers-reduced-motion)` con selector `*` se elimina del bundle minificado. Para reglas globales de a11y usar `<style is:inline>`.
- **Sharp** es excelente para generar favicons y OG images al vuelo desde SVG.
