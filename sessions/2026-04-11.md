### Push y rebase forzoso por divergencia
Al pushear vino la primera sorpresa: el remote rechazó porque el OAuth token de gh no tenía scope `workflow` (lo necesita para crear `.github/workflows/ci.yml`). Resolución: `gh auth refresh -s workflow` con device flow.

Segunda sorpresa: `gh pr create` falló con "branch has no history in common with main". Investigación: el `origin/main` tiene los mismos mensajes de commit pero SHAs distintos al `main` local (`5d15e2c1` vs `406f55d2`) — la history del remote fue reescrita en algún punto (probablemente filter-repo para limpiar archivos grandes), sin merge base. Solución:
1. Crear backup branch.
2. `git rebase --onto origin/main main` → 8/8 commits aplicados sin conflictos (los archivos eran idénticos a pesar del SHA distinto).
3. `git push --force-with-lease` para sustituir el branch remoto con la versión rebaseada.
4. Verificar lint/build post-rebase: ✅ ✅.

### Aprendizajes adicionales (parte 2)

- **Truco "rename puro + modificación"**: cuando `prettier --write` ya divergió los archivos del original, git no detecta renames (umbral 50% por defecto). Para preservar history: extraer HEAD content con `git archive HEAD <path> | tar -x`, sustituir el contenido divergido temporalmente, commit del rename puro, restaurar el contenido divergido, segundo commit con las modificaciones. Resultado: 86 + 932 renames detectados limpiamente.
- **Token gh con scope `workflow`**: NO incluido por defecto en `gh auth login`. Hay que `gh auth refresh -s workflow` antes de pushear cualquier rama que toque `.github/workflows/`.
- **Repos con history reescrita**: el remote `main` y el local `main` pueden divergir si alguien corrió filter-repo. `git merge-base` vacío es la señal. Solución: rebase --onto al remote, no merge.
- **`git push --force-with-lease`** > `--force` siempre que sea posible (verifica que el remote no haya cambiado entre tu fetch y tu push).
